# Marp Presentation Player - Project Memory

## Project Overview
A professional Markdown-to-Slides presentation player built with Next.js 14 and React 18. Users can paste/input Markdown content with Marp syntax, view it as slides in real-time, and manage presentations.

## Technology Stack
- **Framework**: Next.js 14 with App Router
- **Frontend**: React 18 (client components for interactivity)
- **Styling**: Tailwind CSS 3 + Custom CSS for slide themes
- **Markdown Parser**: `marked` library
- **UI Components**: shadcn/ui + custom components
- **Cloud Storage**: AWS S3 (for JSON file uploads/downloads)
- **Deployment**: Docker + docker-compose for containerization

## Core Features Implemented
1. **Markdown Parser**: Parses markdown by "---" slide separator, extracts speaker notes
2. **Live Editor**: Textarea for real-time markdown input
3. **Slide Viewer**: Displays current slide with HTML rendering
4. **Theme System**: 4 built-in themes (Modern, Minimal, Dark, Light)
5. **Navigation**: Previous/Next/First/Last buttons + keyboard shortcuts
6. **Keyboard Shortcuts**: Arrow keys (prev/next), Space (next), Home (first), End (last), F (fullscreen)
7. **Fullscreen Mode**: Dedicated fullscreen presentation view
8. **Speaker Notes**: Collapsible speaker notes section below slides
9. **Save/Load**: JSON export/import for presentations
10. **Play/Pause Slideshow**: Auto-advance slides with configurable intervals (1s-10s)
11. **Slide Animations**: Fade-in and zoom-in animations on slide changes
12. **Responsive Design**: Works on desktop and tablet

## Rendering Pipeline
- **Marp rendering is done server-side** via `/api/render-marp` using `@marp-team/marp-core`
- The API receives raw markdown, renders it with Marp, and returns individual slides as SVG-wrapped HTML
- Each slide HTML includes the full `div.marpit > svg > foreignObject > section` structure required by Marp CSS
- Marp CSS uses highly specific selectors (e.g., `div.marpit > svg > foreignObject > section`) — the HTML structure must be preserved exactly
- The `use-presentation` hook calls the API with 500ms debounce to avoid excessive requests
- Slide viewer renders pre-rendered HTML via `dangerouslySetInnerHTML`

## API v1 (AI Agent / MCP Integration)
- **Database**: PostgreSQL via Prisma. Two models:
  - `SharedPresentation` — stored shared presentations, with optional FK `apiKeyId` to `AppApiKey`
  - `AppApiKey` — user-generated API keys (see "API Key System" below)
- **Auth priorities** (in `lib/api-auth.ts`, `validateApiKey` is async):
  1. Bearer token matching `mkp_<keyId>.<secret>` against `AppApiKey` table (sha256 comparison)
  2. Legacy: comma-separated list in `MARP_API_KEYS` env var
  3. Open mode if neither is configured AND no DB keys exist
- **Endpoints**:
  - `POST /api/v1/render-pdf` — Marp Markdown → PDF binary
  - `POST /api/v1/play` — Store markdown, return play URL + embed URL. Also triggers fire-and-forget `cleanupUnusedApiKeys(30)` + expired presentation cleanup.
  - `POST /api/v1/keys` — Generate a new API key. Requires `acknowledgedSocials: true` in body. Returns full Bearer token **once** (server stores only sha256 hash).
  - `GET /api/v1/presentations/{id}` — Get stored presentation
  - `DELETE /api/v1/presentations/{id}` — Delete presentation
  - `GET /api/v1/openapi` — OpenAPI 3.1 spec (includes `/keys` endpoint with no auth required)
  - `GET /api/v1` — API index
  - `GET /api/config/social` — public config (LinkedIn & GitHub repo URLs from env)
- **Play page**: `/play/[id]` — fullscreen-capable viewer for shared presentations
- **Internal API**: `/api/internal/play/[id]` — no-auth endpoint for play page to fetch data

## API Key System (generated via MCP modal)
- **Token format**: `mkp_<keyId(24)>.<secret(48)>` where keyId and secret are base62 random strings.
- **Storage**: only `keyId` (unique, indexed), `sha256(secret)`, `secretPrefix` (first 8 chars) are persisted. Plain secret is never stored.
- **MCP modal UI** (`components/mcp-modal.tsx`):
  - Placeholder `YOUR_API_KEY` in Claude config + cURL snippets is replaced with the real token after generation
  - "Generate API key" section requires **two checkboxes** — LinkedIn follow + GitHub star — before the generate button unlocks
  - Social URLs come from env vars `LINKEDIN_PERS_URL` and `GITHUB_CURRENT_REPO`, fetched via `/api/config/social`
  - After generation, token is shown **once** with an amber warning banner, reveal/hide toggle, and "I've saved it" acknowledgement checkbox before the user can generate another
- **Monthly cleanup**: `cleanupUnusedApiKeys(30)` in `lib/api-auth.ts` deletes AppApiKey records that have no linked SharedPresentation AND whose `lastUsedAt` (or `createdAt` if never used) is older than 30 days. Called fire-and-forget on every `/api/v1/play` and `/api/v1/keys` POST.
- **FK tracking**: Every presentation created by an authenticated key has `apiKeyId` set, which protects the key from cleanup and lets us build per-key analytics later.

## Key Technical Decisions
1. **Client-side Markdown Parsing**: `marked` library for performance
2. **Manual HTML Rendering**: Regex-based markdown-to-HTML conversion for simplicity
3. **CSS-based Themes**: Theme classes applied dynamically (not Tailwind variants)
4. **Local JSON Export**: Client-side file download (no server processing)
5. **Server Components by Default**: Only client components where needed
6. **Hydration Safety**: All dynamic content in useEffect or client-marked components

## User Preferences
- Professional, clean UI (not gamified)
- Focus on functionality over visual complexity
- Responsive but desktop-first design
- Dark mode support via next-themes
