# Marp Presentation Player - Project Memory

## Project Overview
A professional Markdown-to-Slides presentation player built with Next.js 14 and React 18. Users can paste/input Markdown content with Marp syntax, view it as slides in real-time, and manage presentations.

## Technology Stack
- **Framework**: Next.js 14 with App Router
- **Frontend**: React 18 (client components for interactivity)
- **Styling**: Tailwind CSS 3 + Custom CSS for slide themes
- **Markdown Parser**: `marked` library
- **UI Components**: shadcn/ui + custom components
- **Cloud Storage**: AWS S3 (for JSON file uploads/downloads)
- **Deployment**: Docker + docker-compose for containerization

## Core Features Implemented
1. **Markdown Parser**: Parses markdown by "---" slide separator, extracts speaker notes
2. **Live Editor**: Textarea for real-time markdown input
3. **Slide Viewer**: Displays current slide with HTML rendering
4. **Theme System**: 4 built-in themes (Modern, Minimal, Dark, Light)
5. **Navigation**: Previous/Next/First/Last buttons + keyboard shortcuts
6. **Keyboard Shortcuts**: Arrow keys (prev/next), Space (next), Home (first), End (last), F (fullscreen)
7. **Fullscreen Mode**: Dedicated fullscreen presentation view
8. **Speaker Notes**: Collapsible speaker notes section below slides
9. **Save/Load**: JSON export/import for presentations
10. **Play/Pause Slideshow**: Auto-advance slides with configurable intervals (1s-10s)
11. **Slide Animations**: Fade-in and zoom-in animations on slide changes
12. **Responsive Design**: Works on desktop and tablet

## Rendering Pipeline
- **Marp rendering is done server-side** via `/api/render-marp` using `@marp-team/marp-core`
- The API receives raw markdown, renders it with Marp, and returns individual slides as SVG-wrapped HTML
- Each slide HTML includes the full `div.marpit > svg > foreignObject > section` structure required by Marp CSS
- Marp CSS uses highly specific selectors (e.g., `div.marpit > svg > foreignObject > section`) — the HTML structure must be preserved exactly
- The `use-presentation` hook calls the API with 500ms debounce to avoid excessive requests
- Slide viewer renders pre-rendered HTML via `dangerouslySetInnerHTML`

## Field-level encryption (CRITICAL — see lib/crypto.ts + lib/db.ts)
- **Scope**: only `SharedPresentation.markdown` is encrypted at rest. `title`,
  `apiKey`, etc. stay in clear. Rolling out encryption to more columns means
  adding another hook block in `lib/db.ts` — not touching route handlers.
- **Scheme**: AES-256-CBC with a static IV. Fine for large markdown blobs but
  MUST NOT be repurposed for passwords, short tokens, or anything where
  deterministic ciphertext equality is a leak.
- **Prefix marker `enc:`**: every value produced by `encryptField` is stored as
  `enc:<base64>`. Values without the prefix are treated as legacy plain-text
  and passed through by `decryptField` — this gives us a zero-downtime lazy
  migration path. Do NOT remove the prefix or change it without also shipping
  a data migration.
- **Idempotency**: `encryptField` is a no-op on already-encrypted input, so
  double-encrypting is impossible even if a caller accidentally passes
  ciphertext back into a write.
- **Prisma integration**: `lib/db.ts` uses `PrismaClient.$extends({ query: ... })`
  with hooks on `sharedPresentation` for `create`, `createMany`, `update`,
  `updateMany`, `upsert`, `findUnique`, `findFirst`, `findMany`. Every call
  site (`/api/v1/play`, `/api/v1/mcp`, `/api/v1/presentations/[id]`,
  `/api/internal/play/[id]`, ...) inherits the behaviour for free.
- **Known quirk**: write-side ops (`create`/`update`) return ciphertext in
  `.markdown` because the extension read-hook runs only on read-side queries.
  No current route consumes `.markdown` from a write result; if you add one,
  re-read via `findUnique` or call `decryptField` manually.
- **Env vars** (required in every environment, including dev):
  - `ENCRYPTION_KEY` — 64 hex chars (32 bytes, AES-256 key)
  - `ENCRYPTION_VECTOR` — 32 hex chars (16 bytes, CBC IV)
  Generate with `node -e "const c=require('crypto'); console.log(c.randomBytes(32).toString('hex')); console.log(c.randomBytes(16).toString('hex'))"`.
  Store in hosting secrets (Abacus Settings → Environment / Vercel / Railway /
  Doppler), never commit real values. `.env.example` documents the shape.
- **Ops tooling**:
  - `scripts/encrypt-markdown-fields.ts` — optional one-shot migration that
    re-encrypts any plain-text rows still in the DB. Run with
    `yarn tsx --require dotenv/config scripts/encrypt-markdown-fields.ts`.
  - `scripts/test-crypto.ts` — pure unit smoke-tests (no DB). Verifies
    prefix detection, idempotency, round-trip on unicode, corruption-raises.
  - `scripts/test-crypto-prisma.ts` — end-to-end round-trip through the
    wrapped Prisma client, verifying the DB actually stores `enc:<base64>`.
- **Key rotation**: not automated. When rotating, add an `ENCRYPTION_KEY_V2`
  env var, change the prefix to `enc:v2:`, and re-encrypt rows via a dedicated
  migration script before retiring the old key.

## Authentication model (CRITICAL — do not merge UI and API paths)
Three integration surfaces, three different auth rules, **same** storage:

| Surface                     | Endpoint                 | Auth                                              | `source` tag |
|-----------------------------|--------------------------|---------------------------------------------------|--------------|
| Web UI "Share" button       | `POST /api/internal/play`| Same-origin check (Origin/Referer must equal deployment) | `"ui"`  |
| REST / OpenAPI clients      | `POST /api/v1/play`      | Bearer token (DB-backed `AppApiKey` OR legacy `MARP_API_KEYS`) | `"api"` |
| MCP JSON-RPC `create_share_link` | `POST /api/v1/mcp`  | Bearer token (same as REST)                       | `"mcp"`      |

Rationale:
- End users clicking Share must **not** be asked for a token. The same-origin
  check (browsers always emit `Origin` on POST, cannot be forged by JS) is
  sufficient CSRF protection for this low-impact action.
- Bots / agents / MCP clients **must** supply a Bearer token so usage is
  auditable per-key via `AppApiKey.usageCount` and per-presentation via
  `SharedPresentation.apiKeyId`.
- The `source` column on `shared_presentations` (added in the Share-bug fix
  iteration) is the one knob to answer "who created this?" without joining
  anything. Add an `@@index([source])` is already in place. If you add a new
  surface, assign it its own `source` value (don't re-use `"ui"/"api"/"mcp"`).

`POST /api/internal/play` also enforces:
- 256 KiB hard cap on `markdown` (reject 413 above that — pushes heavy payloads
  to `/api/v1/play` where callers have an identity we can rate-limit).
- Max 1-year `expiresInHours` to prevent runaway retention via a browser console.

If you ever need to change the UI-vs-API split (e.g. add a UI signup flow),
update this table **and** the `source` enum semantics together. Do not point
the UI at `/api/v1/play` again — users will see "Missing Authorization header"
errors and file bugs.

## API v1 (AI Agent / MCP Integration)
- **Two integration paths, same auth, same storage**:
  1. **Native MCP server** — `POST /api/v1/mcp`, JSON-RPC 2.0 over the MCP
     Streamable HTTP transport. For Claude Desktop, Cursor, Cline, etc.
  2. **REST + OpenAPI 3.1** — for GPTs Actions, LangChain, LlamaIndex, etc.
- **Database**: PostgreSQL via Prisma. Two models:
  - `SharedPresentation` — stored shared presentations, with optional FK `apiKeyId` to `AppApiKey`
  - `AppApiKey` — user-generated API keys (see "API Key System" below)
- **Auth priorities** (in `lib/api-auth.ts`, `validateApiKey` is async; shared by MCP and REST):
  1. Bearer token matching `mkp_<keyId>.<secret>` against `AppApiKey` table (sha256 comparison)
  2. Legacy: comma-separated list in `MARP_API_KEYS` env var
  3. Open mode if neither is configured AND no DB keys exist
- **Endpoints**:
  - `POST /api/v1/mcp` — **Native MCP server (Streamable HTTP)**. Implements
    `initialize`, `tools/list`, `tools/call`, `ping`, `notifications/*`. Exposes
    four tools: `create_share_link`, `render_pdf`, `get_presentation`,
    `delete_presentation`. Lives in `app/api/v1/mcp/route.ts` — **do not** point
    MCP clients at `/api/v1/openapi`, which is GET-only and will return 405.
  - `POST /api/v1/render-pdf` — Marp Markdown → PDF binary (REST)
  - `POST /api/v1/play` — Store markdown, return play URL + embed URL. Also triggers fire-and-forget `cleanupUnusedApiKeys(30)` + expired presentation cleanup.
  - `POST /api/v1/keys` — Generate a new API key. Requires `acknowledgedSocials: true` in body. Returns full Bearer token **once** (server stores only sha256 hash).
  - `GET /api/v1/presentations/{id}` — Get stored presentation
  - `DELETE /api/v1/presentations/{id}` — Delete presentation
  - `GET /api/v1/openapi` — OpenAPI 3.1 spec (documents MCP endpoint too, but the spec itself is only a descriptor — *not* an MCP server).
  - `GET /api/v1` — API index (now lists both `mcp` and `openApiSpec`)
  - `GET /api/config/social` — public config (LinkedIn & GitHub repo URLs from env)
- **Play page**: `/play/[id]` — fullscreen-capable viewer for shared presentations
- **Internal API**: `/api/internal/play/[id]` — no-auth endpoint for play page to fetch data
- **MCP modal (`components/mcp-modal.tsx`)**: shows **two** URLs — the MCP
  server URL (`/api/v1/mcp`) and the OpenAPI spec URL (`/api/v1/openapi`).
  The Claude Desktop config snippet uses the MCP URL with `"type": "http"`.
  Do not collapse these two back into one — they serve different client types.

## API Key System (generated via MCP modal)
- **Token format**: `mkp_<keyId(24)>.<secret(48)>` where keyId and secret are base62 random strings.
- **Storage**: only `keyId` (unique, indexed), `sha256(secret)`, `secretPrefix` (first 8 chars) are persisted. Plain secret is never stored.
- **MCP modal UI** (`components/mcp-modal.tsx`):
  - Placeholder `YOUR_API_KEY` in Claude config + cURL snippets is replaced with the real token after generation
  - "Generate API key" section requires **two checkboxes** — LinkedIn follow + GitHub star — before the generate button unlocks
  - Social URLs come from env vars `LINKEDIN_PERS_URL` and `GITHUB_CURRENT_REPO`, fetched via `/api/config/social`
  - After generation, token is shown **once** with an amber warning banner, reveal/hide toggle, and "I've saved it" acknowledgement checkbox before the user can generate another
- **Monthly cleanup**: `cleanupUnusedApiKeys(30)` in `lib/api-auth.ts` deletes AppApiKey records that have no linked SharedPresentation AND whose `lastUsedAt` (or `createdAt` if never used) is older than 30 days. Called fire-and-forget on every `/api/v1/play` and `/api/v1/keys` POST.
- **FK tracking**: Every presentation created by an authenticated key has `apiKeyId` set, which protects the key from cleanup and lets us build per-key analytics later.

## Key Technical Decisions
1. **Client-side Markdown Parsing**: `marked` library for performance
2. **Manual HTML Rendering**: Regex-based markdown-to-HTML conversion for simplicity
3. **CSS-based Themes**: Theme classes applied dynamically (not Tailwind variants)
4. **Local JSON Export**: Client-side file download (no server processing)
5. **Server Components by Default**: Only client components where needed
6. **Hydration Safety**: All dynamic content in useEffect or client-marked components
